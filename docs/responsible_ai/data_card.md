# Data Card — I³ Training and Evaluation Datasets

*Follows the "Datasheets for Datasets" structure of Gebru et al. 2021
(arXiv:1803.09010), adapted for three related datasets used in the I³
prototype: the synthetic interaction dataset used to train the TCN
encoder, the DailyDialog + EmpatheticDialogues subsets used to train the
SLM, and the valence lexicon used for sentiment features in the
perception layer.*

Honesty statement up front: none of the datasets here are novel
collections. The synthetic dataset is generated from published
keystroke-dynamics archetypes. The two dialogue corpora are public
academic releases reused under their stated licences. The lexicon is
compiled by hand from common English valence words and is not a
redistribution of NRC or VADER; this is explicit in §3.

---

## 1. Synthetic Interaction Dataset (TCN training)

### Motivation

**For what purpose was the dataset created?** To train and evaluate a
temporal convolutional network that maps a streaming sequence of
interaction feature vectors to a 64-dim user-state embedding. No
public dataset exists that pairs keystroke-level timing with
archetype labels at the scale and control needed for contrastive
pretraining, so one was generated from published parameter
distributions.

**Who created it?** The project author, Tamer Atesyakar, as part of
the I³ prototype build.

**Who funded the creation?** Unfunded; prepared in the author's own
time for the Huawei London HMI Lab interview.

### Composition

**What do the instances represent?** Each instance is a simulated
*session* — a sequence of per-message 32-dim `InteractionFeatureVector`
frames. The 32 features are grouped 8 × 4: keystroke dynamics, message
content, session dynamics, deviation metrics. See
`i3/interaction/types.py` for the canonical schema.

**How many instances?** 10 000 sessions. Mean session length 18 messages
(range 5–40).

**What sampling strategy?** Each session draws a primary archetype
uniformly from the 8 archetypes listed below, then evolves through a
Markov transition matrix over archetypes at each message step. The
transition matrix is hand-designed to favour diagonal (stay) with
small off-diagonal mass to adjacent-style archetypes (fatigued ↔
distracted, stressed ↔ accessibility-need, etc.).

**The 8 archetypes.** Drawn from the HCI literature on
keystroke-dynamics affect and state detection:

| Archetype                | Primary source(s)                            | Salient features                                        |
|:-------------------------|:---------------------------------------------|:--------------------------------------------------------|
| Normal / baseline        | Epp et al. 2011                              | Moderate IKI, low correction rate                       |
| Fatigued                 | Vizer 2009; Epp et al. 2011                  | Slow IKI, many pauses, short messages                   |
| Energetic / engaged      | Epp et al. 2011                              | Fast IKI, long bursts, long messages, rich vocabulary   |
| Distracted               | Vizer 2009                                   | Long pauses, normal correction rate                     |
| Stressed                 | Vizer 2009                                   | Short bursts, elevated correction rate                  |
| Accessibility-need       | Zimmermann 2014 (soft proxy)                 | Very slow IKI, many backspaces, short fragments         |
| Cognitively loaded       | Epp et al. 2011                              | Slow IKI + short messages + elevated corrections        |
| Expert / fluent          | Zimmermann 2014                              | Fast IKI, rare corrections, high vocabulary diversity   |

Each archetype has a parameter distribution over 32 feature channels
calibrated against the means and standard deviations reported in the
source papers, with Gaussian noise added per frame for within-session
variation.

**Are there missing values?** No. Every feature channel is populated
by construction.

**Does the dataset contain confidential information or data that might
be offensive?** No. The dataset contains no text, no user identifiers,
and no behavioural data from real individuals.

**Does the dataset relate to people?** Indirectly. The parameter
distributions are derived from studies of human participants, but no
individual-level data is reproduced.

### Collection Process

**How was the data acquired?** Procedurally generated by
`training/generate_synthetic.py`. Runs with a fixed seed for
reproducibility.

**Time frame of collection.** April 2026. Single deterministic
generation run.

**Was ethical review obtained?** Not applicable — no human subjects
involved.

### Preprocessing / Cleaning / Labelling

**Was any preprocessing done?** Features are z-score normalised against
per-archetype means and standard deviations before being used for
contrastive training. The raw unnormalised features are preserved
alongside the normalised features so that evaluation can be done in
either space.

**Are all labels and metadata provided?** Yes — every frame is tagged
with the generating archetype, the position within the session, and
the session identifier.

### Uses

**What tasks has the dataset been used for?** Training and evaluating
the TCN user-state encoder via NT-Xent contrastive loss (Chen et al.
2020). Evaluation via silhouette score, KNN top-1, and PCA 2D
separability.

**Other potential uses?** Sanity-checking any streaming feature
encoder. Not suitable for supervised archetype classification in a
production sense — the archetypes themselves are constructs, not
ground-truth user states.

**What should the dataset not be used for?** Any claim about real-world
user behaviour. The dataset is a training vehicle, not evidence.

### Distribution

The dataset is not distributed as a binary artefact. It is regenerated
from the script at any commit via `make generate-data`. This is
deliberate: committing the binary would add ~150 MB of random-process
output that re-generating deterministically is cheaper.

### Maintenance

**Who maintains the dataset?** The project author.

**Will the dataset be updated?** Not planned. If the archetype set
changes (e.g. split the accessibility-need archetype into motor vs.
cognitive sub-types), the generator will be versioned.

## 2. DailyDialog + EmpatheticDialogues (SLM training)

### Motivation

**For what purpose was the dataset created?** The two public corpora
were independently collected to study everyday dialogue (DailyDialog,
Li et al. 2017, arXiv:1710.03957) and empathetic-response generation
(EmpatheticDialogues, Rashkin et al. 2019, arXiv:1811.00207). The I³
prototype reuses filtered subsets to train a small from-scratch
transformer with cross-attention conditioning.

**Who created the originals?** The cited authors and their
institutions.

**Who funded the originals?** DailyDialog: Tsinghua and partner
institutions. EmpatheticDialogues: Facebook AI Research.

### Composition

**DailyDialog.** ~13 K multi-turn English dialogues on everyday
topics. Each turn is tagged with a dialogue act and an emotion label
(neutral / anger / disgust / fear / happiness / sadness / surprise).
For I³ training we use turns as context-response pairs and discard the
act/emotion labels — the cross-attention conditioning is driven by
the user-state embedding, not by those labels.

**EmpatheticDialogues.** ~25 K single-turn English exchanges grounded
in an emotional situation. A speaker sets up a situation ("I got my
first job yesterday!"), a listener replies. 32 emotion categories
cover the distribution.

**Are there missing values?** Minimal; both corpora are well-curated.
A small number of turns in each fail a length or encoding check and
are dropped during preprocessing.

**Does the dataset relate to people?** Yes — both corpora are human-
generated dialogue. Licences are respected (see Distribution below).

**Demographic bias.** Both corpora are English-only. DailyDialog is
described by its authors as covering "10 topics of daily
conversation" with no demographic sampling strategy; the authors have
not released demographic metadata. EmpatheticDialogues was collected
via Mechanical Turk with workers required to be fluent English
speakers; published demographic information is limited but the worker
pool skews US-based. Both corpora are therefore English-, adult-, and
western-cultural-reference-frame-dominant. This is acknowledged in
`model_card_slm.md` and is a known limitation.

### Collection Process

**How was the original data acquired?** DailyDialog: scraped and
filtered from English-learning websites. EmpatheticDialogues:
solicited from Mechanical Turk workers in pairs (speaker / listener).

**Ethical review.** Both original collections describe their
human-subjects handling in their papers. I have not re-run any
consent process; I reuse the released data under the original
licences.

### Preprocessing / Cleaning / Labelling

**Preprocessing applied in I³:**

- Lowercase.
- Punctuation split from adjacent tokens (so `"hello!"` → `["hello", "!"]`).
- Speaker-turn boundaries preserved as explicit turn markers.
- Maximum sequence length 256 tokens; longer turns truncated at a
  token boundary.
- Filter: drop turns shorter than 3 tokens or longer than 256 tokens
  after tokenisation; drop turns with non-ASCII character ratio
  > 10 %.
- Train / val split: 90 / 10, deterministic with seed 42.
- An additional **50-example adaptation-fidelity set** was
  hand-authored by the project author for evaluation only. Each entry
  is a prompt paired with a target adaptation state (cognitive load,
  formality, warmth, accessibility) and a reference human reply. This
  set is **not** used for training.

### Uses

**What tasks has the dataset been used for in I³?** Training a
~6.3 M-parameter Pre-LN transformer with cross-attention conditioning
on a user-state embedding. Teacher-forced next-token prediction with
cross-entropy loss and cosine-warmup learning-rate schedule.

**Out-of-scope uses.** Do not use the filtered I³ subset to evaluate
generic language-modelling claims; the filtering is tuned to the I³
SLM's architecture and context budget and is not representative of
the original corpora.

### Distribution

The original corpora are distributed under the authors' original
licences. The I³ preprocessing pipeline is open and can be run on a
fresh download. The preprocessed tensors are not redistributed to
avoid introducing licensing ambiguity.

DailyDialog licence: CC BY-NC-SA 4.0 (non-commercial, share-alike).
The I³ prototype is a non-commercial research project and the derived
models are released under MIT with a note that the SLM's training
data includes a CC BY-NC-SA 4.0 component; any commercial derivative
must replace the training data or obtain a commercial licence from
the DailyDialog authors.

EmpatheticDialogues licence: Creative Commons Attribution 4.0.

### Maintenance

The project author maintains the I³ preprocessing pipeline; the
original corpora are maintained by their respective authors.

## 3. Valence Lexicon (~365 words)

### Motivation

**For what purpose was the lexicon created?** To provide a lightweight
English sentiment-valence signal for the per-message feature
extraction in `i3/interaction/linguistic.py` without pulling in a
heavy NLP dependency (NLTK, spaCy, transformers).

**Why not reuse NRC or VADER verbatim?** NRC Emotion Lexicon
(Mohammad & Turney 2013) is released under terms that require
redistribution permission; VADER (Hutto & Gilbert 2014) is an
accepted research benchmark but carries its own licence. Rather than
redistribute either verbatim, the I³ lexicon is hand-compiled from
common English valence words and is much smaller (~365 entries
versus NRC's ~14 000). The compilation overlaps conceptually with
published lexicons — there is no way to pick the 50 most
unambiguously positive English words without overlap with NRC —
but no file is copied, no list is redistributed, and the size is
order-of-magnitude smaller.

### Composition

- ~185 positive words, each with a numeric valence score in [0, 1].
- ~180 negative words, each with a numeric valence score in [-1, 0].
- ~15 negation markers ("not", "never", "no", "n't", etc.).
- ~30 intensifiers ("very", "extremely", "super", "really", etc.).

All entries are lowercase English tokens; no multi-word expressions.

**Selection criteria.** Entries were chosen for unambiguous valence
across written English contexts. Ambiguous words ("sharp",
"critical", "sick") are excluded. Domain-specific valence ("bullish"
in finance, "lossy" in engineering) is excluded. The lexicon is
intended for everyday-dialogue sentiment, not sentiment over
specialist corpora.

### Collection Process

**How was the data acquired?** Compiled by hand by the project author
across two sittings in April 2026, cross-referenced against common
English valence conventions but without copying from any single
source.

### Preprocessing / Cleaning / Labelling

Lowercase, no lemmatisation. Negation handling is done at feature-
extraction time (within a 3-token window), not baked into the lexicon
itself.

### Uses

**In I³:** Sentiment-valence feature in the 32-dim
`InteractionFeatureVector` (feature group: linguistic complexity).

**Not recommended for:** Any use outside the narrow per-message
valence signalling described above. The lexicon is small, English-
only, and domain-general; it is not competitive with a trained
sentiment model on benchmarks.

### Distribution

The lexicon lives in the I³ repository (`i3/interaction/linguistic.py`
or a bundled resource file) and is released under MIT with the rest
of the code.

### Maintenance

The project author maintains the lexicon. Corrections and additions
are welcome. The lexicon is not intended to grow significantly beyond
its current scope; if a real deployment needs more coverage, the
right answer is to integrate a proper sentiment model, not to expand
this lexicon.

## Cross-Dataset Notes

1. **The TCN encoder is trained on *only* the synthetic dataset.** It
   sees no public dialogue corpus and no lexicon entries.
2. **The SLM is trained on *only* the dialogue subsets.** It sees no
   synthetic interaction features and no lexicon entries.
3. **The lexicon is used *only* at feature-extraction time** on live
   user input. It is not used during training of either model.

These separations are architectural, not incidental: they preserve
the claim that each component is independently evaluable and that
each dataset's biases are confined to the component it shaped.
